End-to-end request flow — from the Angular frontend through the API Gateway to each microservice and back.

1️⃣
Request arrives at API Gateway (:8080)

JwtAuthFilter checks the path. /api/auth/** is whitelisted — no JWT required. Request passes directly through.

2️⃣
Auth Service validates credentials

AuthService looks up user by email, verifies BCrypt hash, then calls JwtUtil.generateToken(userId, role) — userId is the JWT subject, role is a claim.

3️⃣
Events fired via RabbitMQ

On login/register, EmailPublisher publishes an EmailMessage to RabbitMQ. EmailConsumer picks it up and sends the email asynchronously — fully decoupled.

4️⃣
Token stored client-side

Angular stores the JWT in localStorage. The AuthInterceptor automatically attaches Authorization: Bearer <token> to all subsequent requests.

1️⃣
Premium Calculation

PremiumCalculator applies risk-based formula: base rate × coverage factor × frequency multiplier × age adjustment. Returns annual/monthly/quarterly premium.

2️⃣
Policy saved + Audit logged

Policy entity saved with unique POL-YYYYMMDD-XXXXX number. AuditLog records actor, action, fromStatus→toStatus, and reason.

3️⃣
Premium Schedule generated

Based on term months and payment frequency, a full installment schedule is written to the premiums table — each with a due date and PENDING status.

4️⃣
Notification event published

NotificationPublisher.publishPolicyPurchased() sends a PolicyPurchasedEvent to RabbitMQ. NotificationConsumer calls AuthService to get the email, then sends confirmation.

📝 Claim Lifecycle Flow
👤
Customer
Files claim
Feign: validate policy
📝
Claim Service\n:8083
DRAFT → UNDER_REVIEW
Admin: APPROVE / REJECT
🛡️
Admin Panel
Feign deductCoverage
ClaimDecisionEvent
📧
Email\nNotification
1️⃣
Claim Created (DRAFT)

ClaimService checks policy ownership via Feign (PolicyClient) and verifies the policy is ACTIVE. Validates claim amount ≤ leftoverCoverageAmount.

2️⃣
Document Upload

Customer must upload 3 docs: Claim Form (auto-generated PDF with digital sig), Aadhaar Card, and Evidence. Stored as BLOB in file_data columns.

3️⃣
Submit → Under Review

Customer submits claim. Status transitions: DRAFT → SUBMITTED → UNDER_REVIEW in one call. All 3 docs must be present or exception thrown.

4️⃣
Admin Decision + Coverage Deduction

On APPROVED, Claim Service calls PolicyClient.deductCoverage() with optimistic locking (3 retries). Then publishes ClaimDecisionEvent to RabbitMQ for email notification.

💳 Razorpay Payment Flow
🌐

Razorpay API
razorpayOrderId returned
🏦
Razorpay\nCheckout

1️⃣
Order Creation

Frontend calls POST /api/payment/order. PaymentService uses Razorpay Java SDK to create an order with amount in paise (INR × 100). Returns orderId + keyId.

2️⃣
Razorpay Checkout

Angular opens Razorpay's checkout modal with the orderId. User completes payment through their bank/UPI. Razorpay returns paymentId + signature.

3️⃣
HMAC-SHA256 Verification

Utils.verifyPaymentSignature() validates orderId|paymentId hash against the Razorpay secret. Only verified payments are persisted as SUCCESS.

4️⃣
Policy premium marked PAID

After verification, Angular calls PolicyService.payPremium() with the transactionId as the paymentReference. Premium status updated, PremiumPaidEvent fired.

🔒 Security Architecture

Security Flow Diagram

Multi-layer defense — JWT at the gateway, header propagation to services, internal secret for service-to-service calls, and BCrypt at rest.

Layer 1 — Gateway JWT Filter
🌐
Client Request
Authorization: Bearer <JWT>
✔ JWT valid?
🔀
JwtAuthFilter
HMAC-SHA256 verify · Extract userId + role
jwtUtil.validateToken(token)
extractUserIdAsLong(token)
extractRole(token)
✗ Invalid → pass-through (unauthenticated)
↓ Inject custom headers into mutated request
Layer 2 — Downstream Header Injection
X-User-Id Long userId from JWT subject
X-User-Role CUSTOMER or ADMIN
X-Internal-Secret Shared-secret for inter-service calls
↓ Each microservice reads injected headers
Layer 3 — Service-Level Filters (all microservices)
InternalRequestFilter
Guards /internal/** endpoints. Checks X-Internal-Secret matches configured value. Returns 403 if missing or wrong.
Applies to: Auth, Policy, Admin, Payment, Claim
HeaderAuthenticationFilter
Reads X-User-Id + X-User-Role headers. Creates UsernamePasswordAuthenticationToken with ROLE_CUSTOMER/ROLE_ADMIN and sets it in SecurityContext.
Enables method-level @PreAuthorize checks
Method Security
@EnableMethodSecurity on all services. Controllers use @PreAuthorize("hasRole('ADMIN')") or check role programmatically via SecurityUtils.
Ownership checks in PolicyService, ClaimService
↓ Data-level security
Layer 4 — Data & Endpoint Security
🔐
BCrypt Password Hashing
All passwords stored as BCrypt hashes. BCryptPasswordEncoder is the only PasswordEncoder bean — no plaintext ever stored.
⏱️
JWT Expiration
Tokens have configurable expiry via jwt.expiration property. Expired tokens fail validateToken() and are not propagated.
🔒
CSRF Disabled (Stateless)
All services use SessionCreationPolicy.STATELESS. CSRF disabled because token-based auth makes it irrelevant.
🏦
Razorpay HMAC Signature
Payment verification uses Razorpay's Utils.verifyPaymentSignature() with HMAC-SHA256 to confirm payment authenticity before recording.
🔑
Ownership Enforcement
PolicyService and ClaimService verify policy.getCustomerId().equals(currentUserId) before any mutation. Admins bypass via role check.
🔄
Optimistic Locking
Policy entity uses @Version for optimistic locking. deductCoverage() retries up to 3 times on ObjectOptimisticLockingFailureException.

🏗️ Key Design Decisions

  • InternalRequestFilter — prevents external clients from calling /internal/** by checking X-Internal-Secret header
  • HeaderAuthFilter — stateless; trusts headers injected by API Gateway

  • API Gateway uses Eureka for load-balanced routing via service names
  • Startup order: Config → Eureka → Gateway → Services (enforced by Docker health checks)

🏗️ Startup Orchestration

  • MySQL health check → Config Server starts
  • Config Server health → Eureka starts
  • Eureka health → API Gateway + all microservices start
  • All services additionally depend on MySQL, Redis health
  • Auth and Policy additionally wait for RabbitMQ health
